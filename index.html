Hero project · Shopee

Diana — Skill Engineering for an LLM Data Agent

BI Engineer Intern · Buyer & Marketing BI Team · Feb 2026–Present

Diana is an internal LLM-powered data agent. The platform belongs to another team; I own the application-constraint layer — designing the Skills (runbooks) that let analysts hand recurring AB-test analyses to the agent and trust the numbers it returns.

Manual analysis 30–45 min
Diana Skill 2–3 min

Before: hand-write SQL, switch between two data sources, edit region / date / group_id, run each region, assemble a comparison table. After: fill a form → Skill auto-runs → structured report.

Honest framing: the saving comes from automating the analyst's manual workflow, not from making any single query faster. The Skill makes the agent's reasoning auditable and repeatable.

What I built

  • 3 production Skills: Me-Page traffic, PDP ads-voucher CR, PDP OPU & GMV
  • 5-section runbook structure: inputs → procedure → SQL template → output schema → pre-return checklist
  • Hardcoded invariants & locked denominators over free-form generation
  • A Skill-generator (v1→v2→v3) with 14 lint rules to make authoring self-checking

Why it's reliable

  • Runbook style is followed top-to-bottom; prose style gets partially skipped
  • Enum-constrained outputs (Literal + __post_init__ validation)
  • LEFT JOIN + COALESCE so denominators can't silently shrink
  • Black-box evaluation feeds a 9-mode failure taxonomy back into design

Most dangerous failure — and how I caught it

The silent LEFT JOIN → INNER JOIN collapse

An early SQL referenced a left-joined column in WHERE. When the key was NULL, NULL IN (...) dropped the row — turning the LEFT JOIN into an INNER JOIN, shrinking the denominator and inflating conversion rate. A syntax error screams; a wrong denominator looks perfectly normal and quietly poisons the decision. I made the join invariant explicit so it can't regress.

Project · Shopee

AB-Test Text-to-SQL Workflow

Multi-region traffic & funnel analysis pipeline

A deterministic low-code workflow for AB-test analysis across 8 regions. The key engineering judgment: shrink the LLM's job to Stage 1 (variable extraction) and make everything downstream deterministic — flexible enough to take natural language, reliable enough for production.

Query latency: ~300 s → ~2 s

Raw-table scan ~300 s (timeouts)
Pre-agg mart ~2 s

Earlier filtering, fewer redundant joins, and migrating expensive raw-table queries onto pre-aggregated data marts.

Honest framing: this speedup is data modeling & query optimization (pre-agg marts, predicate pushdown, join control) — not the LLM. The agent automates the analysis chain; it does not accelerate the database.

Project · NTU CA6123

Multi-Agent Literature Monitoring System

Google ADK · Gemini · 4-person team · Apr–May 2026

A multi-agent system that monitors frontier-AI research and produces personalized digests. I owned the orchestrator architecture and the design decisions; I picked up Google ADK from scratch for it.

Architecture (mine)

  • ADK chosen over LangChain/LangGraph after a trade-off study
  • SequentialAgent + LoopAgent orchestrator with a critic-revision loop
  • 2 human-in-the-loop checkpoints: after planning, after synthesis
  • Dual storage: SQLite (state) + ChromaDB (semantic recall)
  • Observability: traced decorator, per-step cost tracking

By the numbers

  • ~3,990 LOC in my modules (orchestrator 642, schemas 498, storage 544, observability 330)
  • 18/18 stdlib smoke tests pass + full pytest
  • 7 test files · ~1,285 LOC of tests
  • 11 dataclass contracts with runtime __post_init__ validation
Per the course acknowledgement, an AI coding assistant generated boilerplate scaffolding. The architecture, ADK composition, schema-design principles, and orchestrator critic loop are my own; PR history documents the decisions.

Project · NTU RL Lab

Hand-Coded Adversarial Agent — "Design beats scale"

Pure Python (stdlib only) · ~600 LOC · May 2026

In a battle-royale RL environment, most teams trained 70M-step neural-net agents. I wrote a code-as-policy agent by hand and, through black-box behavioral analysis and ablation, found the blind spots in the trained NN's lead-the-shot prediction.

Tied 1st on average placement vs a 70M-step trained NN

Tournament — 50 games

Bars = wins. Average placement (more robust to luck in battle royale) is tied: 3.14 / 3.14 / 3.16.

Ablation — win rate vs NN by engage distance

ENGAGE_DIST = 0.22 is where the NN's lead-the-shot becomes unreliable at close range.

The same thesis as Diana, in a different domain: in a narrow regime, domain-specific constraint design often beats scaling generic ML. Not anti-ML — Diana itself is built on an LLM — but "having a model" isn't "solving the problem."

Research

Research Sketch

Early-stage proposal for discussion

When students use AI tools for learning, how does the format of AI-generated feedback — direct explanation vs. scaffolded self-evaluation prompt — affect conceptual retention and metacognitive awareness?

Motivation

Teaching 300+ students, I saw fluent AI explanations help complete tasks while bypassing the cognitive effort needed for durable understanding. If the form and timing of intervention shapes receptivity, can interface design make learners do the cognitive work required to actually learn?

Three Interface Conditions

1Direct Explanation
Complete answer delivered immediately.
2Socratic Scaffolding
Guided questions before revealing answer.
3Self-Evaluation First
Learner predicts & rates confidence first.

Measures

Conceptual RetentionImmediate & delayed post-tests
Metacognitive AccuracyPredicted vs. actual performance
Learning ProcessInteraction logs, time-on-task, help-seeking
User ExperienceEffort, usefulness, frustration, trust

Expected Contribution

An interface design pattern for LLM-based learning tools that balances answer quality with learner reflection, self-evaluation, and durable understanding.
